# OmniPortal Server Etiquette

## Purpose

This note is for future agents touching OmniPortal infrastructure.

Use it to keep the OCI VPS, Raspberry Pi, and deployment flow tidy instead of turning them into a pile of one-off fixes.

## Folder layout on the OCI VPS

We intentionally keep service folders in the home directory for beginner-friendly clarity.

Use this pattern:

```text
~/
  server-services/
    coturn/
    reverse-proxy/
  omniportal/
```

Meaning:

- `server-services/` is for supporting infrastructure
- `omniportal/` is for the actual application stack if it later moves onto the VPS

Do not mix unrelated services into one folder.

Current live OCI service folders:

- `~/server-services/coturn`
- `~/server-services/reverse-proxy`
- `~/server-services/README.md`
- `~/omniportal` reserved for future app-side use

## Docker rule

Default rule:

- if it is an app or support service, run it in Docker
- if it is basic machine administration, leave it on Ubuntu

Examples that belong in Docker:

- coturn
- reverse proxy
- backend
- databases

Examples that do not need Docker:

- SSH
- firewall management
- package manager
- Docker itself

## One folder per service

Each service folder should normally contain:

- `compose.yaml`
- `.env`
- config files like `turnserver.conf`
- optional `data/` folders if the service needs persistent files

That keeps each service understandable in isolation.

## Secrets discipline

Never hardcode live secrets into the repo if a remote `.env` file can hold them instead.

Preferred pattern:

- repo stores code and templates
- server stores live `.env` values

Examples:

- TURN shared secret
- API keys
- tokens

## Ports discipline

Ports are shared across the whole machine even when services live in separate folders.

Before adding a service:

- check what already owns the port
- document the port choice
- avoid casually reusing `80`, `443`, `3478`, or relay port ranges

For coturn, the usual important ports are:

- current deployed main listener: `443/tcp` and `443/udp`
- relay range currently opened: `49160-49200/tcp` and `49160-49200/udp`
- `3478/tcp`
- `3478/udp`
- optional `5349/tcp` for TLS TURN later
- a relay range such as `49160-49200`

## Change etiquette

Before changing a live server:

- inspect first
- prefer reversible changes
- document why the change exists
- keep commands idempotent when possible

Do not:

- scatter manual config across random directories
- leave undocumented background processes
- install app dependencies globally unless there is a strong reason
- create a second deployment path when one already exists

## Pi deployment etiquette

The Pi already has an established deployment flow.

Use:

- [run-pi-backend-deploy.ps1](/D:/PWD/OmniPortal/scripts/run-pi-backend-deploy.ps1)
- [pi-backend-deploy.sh](/D:/PWD/OmniPortal/scripts/pi-backend-deploy.sh)

Important behavior:

- deployment copies current workspace files to the Pi
- it does not require a git pull on every deploy
- the active repo path on Pi is `/home/codex/OmniPortal`

Do not switch future automation back to `/home/pi/OmniPortal`.

## TURN policy

Current intended design:

- browser client and host attempt direct connectivity first
- backend serves direct ICE and relay ICE from `/api/config`
- self-hosted coturn on OCI is the preferred relay path
- public OpenRelay is only a fallback of last resort, not the main design

Prefer short-lived TURN credentials generated by the backend over long-lived TURN usernames/passwords baked into clients.

Current live TURN implementation:

- OCI service path: `~/server-services/coturn`
- service manager: `docker compose`
- backend relay mode on Pi: `turn_rest_env`
- backend mints temporary TURN username/credential pairs from `OMNI_TURN_REST_SECRET`
- coturn uses the same shared secret via `--use-auth-secret`

Useful OCI checks:

```bash
ssh oci "cd ~/server-services/coturn && docker compose ps"
ssh oci "cd ~/server-services/coturn && docker compose logs --tail 40"
ssh oci "sudo ss -lntup | grep ':443'"
```

Current notable quirk:

- coturn logs still show `ERROR: CONFIG: Unknown argument:` at startup
- despite that log line, the service is functioning and was verified reachable on `161.118.184.141:443`
- do not assume the service is broken solely because of that message

## Current validation outcome

Latest known user test result after the self-hosted TURN move:

- connection works

- much better than the earlier shared-relay path

Interpretation:

- promising for future screen sharing and remote control

## When handing off

Leave behind:

- service folder paths
- active ports
- where secrets live
- how to restart each service
- how to verify health

If a future agent can SSH in and understand the box in five minutes, the setup is clean enough.
